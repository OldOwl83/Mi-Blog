Features of this Game Version

This program implements what I understand to be a classic version of the game. This implies applying an old rule that is now unusual: Ships cannot touch each other, so that lockers adjacent to a boat must remain empty. This entails an additional difficulty for the validation of the positions that the various vessels aspire to occupy, both for manual distribution of fleet and for its automatic randomization. Additionally, it offers an overcrowded 10x10 board, on which 15 boats of different sizes must be placed (1x5, 2x4, 3x3, 4x2, 5x1). Thus, 35% of the lockers are occupied during the course of the game, and many others can be ruled out due to its proximity to a discovered ship. So, the style of play is one of a hard "blow for blow", where the bombings of combat are constantly heard, giving rise to agile and dizzying dynamics.

Regarding the behavior of the machine, it acts in a similar way to a human when it hits a shot and pursues the objective of sinking the found ship. However, it is not so smart when it comes to sweeping the board looking for new vessels, as it randomly fires at any untouched positions. This is a great advantage for the experienced player, who tends to win more often than the machine. This aspect could be improved in future versions of this application.

BattleShip is a turn-based game, and although the application allows the user to shoot continuously, anticipating the machine's shots, the final winner will be whoever takes down the entire enemy fleet first on his turn.

About this program

This program is entirely based on HTML5, with pure Javascript, without non-native libraries, frameworks or templates. The whole code was written by me, and even part of its graphic resources (all draws into the boards lockers) were created by me (in fact, they are not spectacular). The audio pieces and decorative images (with the exception of the footer stamp, of course), instead, were taken from the web.

Although the code defines a "Ship" class to record the location and status of each vessel, this program is not object-oriented. The general procedure consists of associating specific functions that manipulate global stored data to certain events (mainly the "click" of mouse and the "drag and drop" of user's ships images), so that the user's action promotes the execution of the different stages of the game. In this sense, the "globals_and_listeners" file would represent the "main" function (without being one, really), since the variables that will store all the "tangible" elements of the game (fleets and boards) are declared and initialized there, and it links the interaction elements of the HTML document with the main specific functions. I think this structure was due both to my unfamiliarity with JavaScript objects when I designed this software, and to the fact that it seemed me more intuitive from a code organization point of view. However, I am aware that this aspect can be extremely improved.

The game's web pages are not responsive in the strict sense. They were intended for a desktop application. However, I have added three stylesheets for the layout to fit different screen resolutions (800-1199px, 1200-1550px, and 1551 and above px wide).

In short, I highlight these defects to emphasize that the effort was mainly put on the efficiency of the code and on the resolution of logical difficulties at the algorithm level. In this regard, the greatest challenges were the validation of legal positions of ships, the building of random distribution of fleets, the record of the ship and shot situations on the boards, the machine's estrategy to sink ships, the synchronization between the user's shooting rhythm and the evaluation of the game turns to award the victory, among other matters.
